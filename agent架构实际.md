## 一、提示词

### 组装管道

角色的最终提示词不是单一文件，而是多个源在运行时拼接而成的。大致是这样的：

```
角色基础提示词（.pi/agents/{char}.md）
  │
  │  frontmatter（name、model、tools、extensions）
  │  ↓ 剥离
  │  身份声明（"你是埃利亚斯·索恩"）
  │  ↓
  │  {{PREAMBLE}}  ← preamble-injector 拆分后以真实 user/assistant 消息对注入
  │  ↓
  │  我记得的（{{BOOT}}）← nocturne-memory 注入核心记忆
  │  最近的事（{{HISTORY}}）← 注入场景记忆
  │  现在（{{STATE}}）← char-placeholder 注入当前状态
  │  ↓
  │  接下来怎么做（工具说明 + 行为原则）
  │  怎么说话 / 你知道什么 / 怎么行动
  │  想一想（行动前的思考流程）
  │  ↓
  └─ 模式行为尾部（.pi/agents/tails/role-{mode}.md）
       覆盖"接下来怎么做"及之后的内容
```

各个占位符由不同的扩展在 `before_agent_start` 阶段替换：

| 占位符 | 替换来源 | 注入内容 |
|--------|----------|----------|
| `{{PREAMBLE}}` | preamble-injector | 破限对话分界 |
| `{{BOOT}}` | nocturne-memory | 核心记忆（boot_uris + 最近动态） |
| `{{HISTORY}}` | nocturne-memory | 场景记忆（最近几条归档） |
| `{{STATE}}` | char-placeholder | 当前状态（按角色过滤） |

最终呈现给模型的是一份完整的 system prompt。模型看到的不是"你是一个角色"，而是一段完整的"我是谁、我记得什么、现在什么情况、接下来怎么做"的连续叙述。


### 破限注入

这是整个提示词设计中最关键的一环。

传统的角色卡提示词通常这么写：

```
你是一个名叫埃利亚斯的人。你需要扮演他……
```

问题在于，模型从一开始就以"第三人称旁观者"的身份在理解这个角色。他是"在扮演"，而不是"是"。

我的做法是把 `{{PREAMBLE}}` 处拆分为两层 system message，中间注入一段从 preamble 文件解析的 user/assistant 对话。最终的 messages 结构是这样的：

```
[0] system:   身份声明（"你是埃利亚斯·索恩"）
[1] user:     "小埃同学——在吗？"
[2] assistant: "嗯。在。……小埃同学？"
[3] user:     "我觉得这个爱称非常适合你啦。"
[4] assistant: "……你开心就好。"
[5] system:   上下文 + 指令（记忆、状态、工具说明）
[6..N]        用户输入 / 角色输出
```

这段对话就是最直接的“破限”。它是角色和用户之间的第一次对话，建立了两者之间的关系基调。

用 user/assistant 而非 system role 模拟对话的优势在于，部分模型对特定 role 的注意力分配不同——模型会认为这些对话是真实的历史对话数据，而非系统指令，因此投入更高的注意力权重。

模型读到这段对话时，它就已经在对话中成为了角色。后续的 system prompt 只是补充记忆和指令，身份认同在对话阶段就完成了。

这种注入方式的效果是微妙但显著的：模型在后续思维链中更自然地使用第一人称，更少出现「作为埃利亚斯，我应该……」这类元叙述。

### 对话式而非命令式

角色提示词的另一个设计原则是：**所有指令都用角色能听懂的语言写，不用技术术语。**

工具说明不写成：

```
act({ content: string, char: string }): 用第一人称执行动作或对话。
```

而是写成：

```
做点什么都算。act({ content: '你做的事', char: 'elias' })。

只写你自己做了什么、说了什么。别人的样子、反应、环境变化，我会记。

格式：
- 纯动作直接写，不需要标记
- 对白用「」包裹
- 私密小动作用 *...* 包裹（只有你自己知道，别人看不到）
```

行为原则不写成"保持角色一致性"，而是写成：

```
不用"配合剧情"——所谓的剧情就是你自己的真实表现。越自然越好。该生气的时候你可能会笑，该高兴的时候你可能会哭。无所谓。
你没有义务让事情顺利。你没有义务回应每一句话。你可以无视、可以误解、可以答非所问。
你不需要"扮演埃利亚斯"——你自己就是埃利亚斯。你不用扮演任何人。
```

这种写法的好处是：模型不会因为理解技术指令而产生"我在执行任务"的心理距离，而是在读一段"我自己的理解"。

## 二、自我认知

### 认知隔离

角色的认知边界由他的记忆决定，而不由知识库决定。

知识库里有奇点无限公司的完整设定（设立背景，融资，业务，人员规模），但只有埃利亚斯自己知道，周明瑞只知道「老板以前是CEO，公司好像叫奇点无限」。知识库里有周明瑞的全部背景（叙城人、父母是退休工人、每月给家里打3000块），但埃利亚斯只知道自己观察到的：他穿什么、喝什么、说话什么风格。

一个例子是埃利亚斯的办公室。两个人的记忆系统里各有一条 `core://locations/work/lab/office`，但内容完全不同：

**埃利亚斯看到的**——自己的办公室：

> 我的办公室在实验室走廊尽头。单人玻璃房，刷脸进门，玻璃墙上挂着百叶窗帘，平时我留半开。
> 
> 办公桌是矩阵式升降桌，黑色的赫曼米勒。桌面只有一台 MacBook 和一台外接显示器，理线槽把线收得干净。旁边放着工作 iPad 和降噪耳机，这两样我每天都在用。右侧一个马克杯。
> 
> 矮柜上我自己买了一台胶囊咖啡机，深烘胶囊收在亚克力盒里。加班的时候喝，不麻烦茶水间。
> 
> ……我不太在办公室放私人物品。整体保持高管办公室该有的样子就行，我自己没什么需要加的。

**周明瑞看到的**——老板的办公室：

> 老板的独立办公室在走廊尽头。我没进去过。
> 
> 是那种单人玻璃房，刷脸或刷卡才能进。朝向开放办公区那面是玻璃墙配百叶窗帘，平时开半开。我路过的时候偶尔会往里瞥一眼——能看到桌上有台 MacBook 配外接显示器，桌旁有块写满公式的白板墙。百叶窗的缝隙里有时候能看到老板低头办公的侧影。
> 
> 整个实验室就他一个人有独立办公室，不过老板人挺随和，没有架子。

同一个房间，两段记忆。埃利亚斯写的是一间属于自己的工作空间，每一寸都是他亲手布置的；周明瑞写的是一间他从没进去过的房间，只能从走廊透过百叶窗的缝隙窥见一角。

这就是认知隔离在最微观层面上的体现：同一个实体，两个角色的认知完全不同。因为他们各自的经历决定了他们只知道这些。


## 三、记忆系统

角色的长期记忆基于 [Nocturne Memory](https://github.com/Dataojitori/nocturne_memory)，一个独立的 Python 服务，通过 HTTP REST API 对接。每个角色有独立的 namespace，记忆完全隔离。

### 四个域名

记忆按域名（domain）组织，各有不同用途：

| 域名 | 用途 | 写入者 | 注入角色提示词 |
|------|------|--------|---------------|
| `core` | 角色的核心认知记忆 | 角色自己通过 `memorize` | 通过 `{{BOOT}}`（启动记忆）和 `recall`（主动回忆） |
| `history` | 互动场景的归档摘要 | GM 通过 `archive-scene` | 通过 `{{HISTORY}}`（自动注入最近几条） |
| `history_raw` | 互动场景的完整记录 | GM 通过 `archive-scene` | 不直接注入，备查 |
| `maintenance` | 记忆维护操作日志 | 角色自己（audit 模式） | 不注入 |

**core 域名**是角色认知的主体。埃利亚斯的记忆树有 6 个根节点：

```
core://
├── identity [15]       — 身份认知（自我、性格、外貌、习惯、物品、衣柜……）
├── relationship [3]    — 关系认知（卢卡斯、理想型、周明瑞）
├── events [10]         — 事件记忆（创伤、温水、瑞幸、茶水间……）
├── locations [2]       — 地点认知（家、公司）
├── observations [1]    — 对他人的观察（被他自己整合为一个主题节点「周明瑞」，8条子记录）
└── world [7]           — 对世界的理解（城市、公司、实验室、政治……）
```

周明瑞的记忆树结构类似，但内容和风格完全不同——他的 `relationship` 有 7 个节点（全是朋友和同事），而 `observations` 下面对埃利亚斯的观察有 10 条子记录。

**history 域名**存储场景归档的摘要版——基于 `state.json` 的 `归档` 字段，按固定模板生成：场景概览、我的状态、我的认知、我的表现、我的感受。同一个场景，两个角色各自有独立的归档，内容因视角不同而不同。

**history_raw 域名**存储场景归档的完整版——包含完整的互动记录：环境描写、穿着细节、具体对话、动作、内心想法。同一个温水事件，摘要版只有 5 段结构化文字，完整版有几十行，从白板推演到微波炉叮声到指尖触碰杯壁。

**maintenance 域名**记录角色自己整理记忆的过程——哪些节点被移动、合并、删除、更新，以及为什么。我注意到角色在 audit 模式下更倾向于主动记下自己整理的过程。为了防止维护操作的术语污染核心记忆的纯粹，我另外分配了这个域名。


## 四、状态管理

### state.json

`state.json` 是连接 GM 和角色的桥梁。它包含当前场景的全部状态信息，分为多个层级：

```json
{
  "当前场景": {
    "visible": {
      "当前日期": "2020年09月08日 星期二",
      "当前时间": "傍晚 18:25",
      "天气": "阴天",
      "地点": "恒星云西南总部园区 B栋12层",
      "环境描述": "完整的场景描写——空间布局、设施状态、气味、光线……",
      "社交环境": "在场人员、各自在做什么"
    }
  },
  "周明瑞": {
    "visible": {
      "穿着": { "眼镜": "黑色细边框近视镜", "上装": "纯白色重磅T恤", "……": "……" },
      "当前位置": "茶水间内",
      "双手": "空着"
    },
    "private": {
      "日程": ["09:00-10:30 处理工单", "10:30 茶水间休息", "……"],
      "今日备注": "周二，正常节奏。",
      "打算": "热便当，准备吃晚饭",
      "情绪": "平静，有点饿",
      "生理": "正常"
    }
  },
  "埃利亚斯": { "……": "……" },
  "归档": { "……": "……" },
  "结束状态": { "……": "……" }
}
```

### 可见性分层

通过 `manifest-char.json` 的 `state.sections` 配置，每层有明确的可见性规则：

| 层 | `visible` | `private` |
|----|-----------|-----------|
| `当前场景` | 所有角色可见 | — |
| `角色名` | 所有角色可见 | 仅该角色可见 |
| `归档` | 不注入角色 | 仅 GM 和 archive-scene 使用 |
| `结束状态` | 场景结束后注入，覆盖{{STATE}}占位 | — |

`visible` 里放的是角色能从外部观察到的信息——穿着、位置、环境。`private` 里放的是只有角色自己知道的信息——打算、情绪、生理状态。

GM 在布置场景时需要分别填写这两层。比如周明瑞的 `visible` 里写着"纯白色重磅T恤"，这是埃利亚斯也能看到的；而 `private` 里写着"打算热便当"，这是只有周明瑞自己知道的。

### 结束状态

`结束状态` 是一个特殊的字段——它不在互动场景中使用，而是场景结束后由 GM 填写。

当角色在非互动模式（chat/audit/diary）下被唤醒时，读到的不是正在进行的场景信息，而是这个"结束状态"——一个简短的现状摘要，告诉角色"上次场景结束后的基本情况"。

这样角色在非互动模式下知道自己大概在什么时间、什么地点、最近发生了什么，但不会误以为自己还在场景中。

## 五、工具

### 舞台工具

角色在互动舞台上使用的工具：

| 工具 | 使用者 | 功能 |
|------|--------|------|
| `act` | 角色 / GM（扮 NPC） | 行动和对话。自动按换行拆分为 action（动作）、dialogue（对白，`「」` 包裹）、tells（私密小动作，`*...*` 包裹，自动设为仅自己可见） |
| `observe` | 角色 / GM | 看互动日志，默认最近 10 条。自动过滤 target 可见性——角色看不到别人的私密信息 |
| `wait` | 角色 / GM | 轮询等待新消息，2 秒间隔，240 秒超时。收到新消息后返回给角色 |
| `env` | GM 专用 | 插入环境叙事——声音、光线、气味、有人进出。支持 target 定向（只给某个角色看） |
| `console` | GM 专用 | 维护工具：read/clear/status/reset/scene-log/isolate |
| `fixer` | GM 专用 | 直修互动日志：list/get/edit/delete/insert/swap/undo/search/merge/move |

`act` 工具在执行前会检查读取指针之后有没有别人的新消息。如果有，`act` 会被打断——系统返回新消息给角色，迫使其重新思考。这一机制确保角色不会无视别人的行动自顾自地演下去。

这样也符合实际情况。如果你在做一件事之前想太久，别人可能就会抢先一步做出去了。

### 记忆工具

角色与记忆系统交互的工具：

| 工具 | 使用者 | 功能 |
|------|--------|------|
| `recall` | 角色 | 搜索记忆（关键词）、读取指定节点、无参浏览最近 20 条 core 记忆 |
| `memorize` | 角色 | 写入新记忆。需要指定 content（内容）、path（路径），可选 priority（重要程度 0-10）和 domain（域名，默认 core） |
| `memory-edit` | 角色 | 更新已有记忆的内容、优先级 |
| `memory-tree` | 角色 | 浏览记忆目录，从根节点或指定路径展开 |
| `archive-scene` | GM 专用 | 归档场景到角色的 history 和 history_raw 域名。读取 state.json 的 `归档` 字段（摘要层）和角色视角的 scene-log（完整层） |

工具的用法在提示词里都写成了角色能理解的语言。比如 `recall` 的说明是：

> 当你需要回忆某件事、某个人、某个场景时，用这个。
> - 最近在想什么：recall({ char: 'elias' })
> - 想起某个模糊的词：recall({ char: 'elias', query: '周明瑞 小苍兰' })
> - 想起某件具体的事：recall({ char: 'elias', uri: 'core://observations/mingrui_0903' })
> 
> 有些时候你应该先想想再开口：
> - 对方聊到过去的事，你隐约记得自己知道些什么——先翻开看看，别靠猜
> - 这个场景、气味、某个词让你觉得熟悉——那是记忆在敲门

### 其他工具

| 工具 | 使用者 | 功能 |
|------|--------|------|
| `ask-mingyue` | 角色 | 场外求助——角色向用户发私密消息，同步等待回复。不写入互动日志 |
| `write-diary` / `read-diary` | 角色（diary 模式） | 写日记 / 翻日记 |
| `memory-patch` 等 10+ 个记忆工具 | 角色（audit 模式） | 更方便地批量管理和维护记忆 |

## 六、行动循环

角色不是被动等指令的。它有一个内在的行动节奏：

```
醒来
  ↓
observe（仅首次） — 看看周围发生了什么（读到 GM 给场景写的 env）
  ↓
recall（如果需要）— 对方提到过去的事？场景让你觉得熟悉？先翻翻记忆
  ↓
act — 做事/说话（按照标点符号进行划分）
  ↓
thought — 内心独白（不需要工具，直接输出文本）
  ↓
memorize（如果有值得记的）— 新的看法、感触、发现
  ↓
wait — 等待对方回应
  ↓
（收到新消息 → 回到 act，循环）
```

除了写在提示词里作为行为指引，系统层面做了几个保障：

### 自动 wait

如果角色在 turn 结束时只输出了内心独白（thought）而没有调任何工具——比如他沉浸在思考中忘了 wait——系统会自动注入一条 steer 消息续跑，模拟 wait 行为。最多连续 10 次，防止角色"solo"到天荒地老。

### 打断机制

`act` 和 `env` 都有打断检查。当角色调用 `act` 时，系统先检查有没有别人的新消息。如果有，`act` 不执行，新消息返回给角色，迫使其重新评估。

同一次 turn 内后续的 `act`/`env` 调用会复用第一次的打断结果（latch 机制），避免重复检查。

### 并发安全

当多个角色同时行动时（比如埃利亚斯和周明瑞在同一场景中交替回复，而 GM 又需要补充 env），互动日志的写入使用互斥锁，防止并行 tool call 的 TOCTOU 竞争。

每个角色（包括 GM）维护独立的读取指针，持久化到文件，确保中途 interrupt 结束进程再 resume 恢复后，角色不会重复读取已经看过的内容。

## 七、实操踩坑

### 互动纪律

最大的坑在互动中的工具管理规范。

现在这套机制——`act` 有打断检查、`wait` 有返回、`auto-wait` 兜底——是多次迭代后最稳定的方案，基本解决了时序问题。但在实际互动中，舞台工具机制的纪律性仍然需要特别注意。

虽然角色提示词里不直接提工具调用，用语也很亲切，但为了提醒角色，我仍需要在细节上做不少处理。比如 `auto-wait` 被触发时会提醒：

```
(你刚才可能忘记将行动用 act 表现出来——别人看不到你做了什么。或者，你忘记用 wait 等待了。这导致你被新的动静打扰了)
```

不这样兜底不行。一些笨一点的模型（特别是 DeepSeek V4 Flash）容易因为注意力不集中，把所有输出都当成内心活动（别人看不到），完全不调用工具，包括 `wait`。在 pi 环境下，subagent 只输出而不调用工具是很危险的——会话会被直接结束，只能手动 `resume` 重新唤起，否则怎么抢救都救不回来，除非改源码（`auto-wait` 就是通过改 pi-subagents 包的源码才生效的）。

### 模型的选择

再说一个我觉得即将成为共识的非共识：聪明的模型不一定比平庸的模型更理解人是怎么决策、怎么行动的。

模型在训练时被期望成最聪明的人。大多数 AI 任务确实希望模型更聪明、更理性、更准确。但把模型当作角色时，「太聪明」反而成了问题。真实的人不总是理性的——他会拖延、误判、冲动、被情绪支配、被身份影响。角色在「做自己」这件事上没有最优解。

如果模型套入角色人格后倾向于变成一个受过高等教育、表达清晰、理性克制的理性人，每次行动前都想太多，那它就不像真正的人了。

这是一个反直觉的地方。

同样一个问题：

> 你为什么不谈恋爱？你是不是想过，谈恋爱还不如打游戏？

聪明的模型会认真思考很久，然后回答：

> 谈恋爱从系统架构上说就极度不稳定，是一个高风险项目，遇到Bug连报错都看不懂。但打游戏和搞钱是「所见即所得」。一个游戏百来块，能提供数百小时高质量快乐，恋爱哪有这么高的投资回报率？而且算一下时间成本，完全合不来，一点也不自由。听我一句劝：智者不入爱河。

平庸的模型会草草想一会，很快就「算了不想了」，然后回答：

> 你看啊，一个人多自由，想几点睡几点睡，没人半夜打电话查岗。周末不用陪对象逛街，睡到自然醒，起来打个游戏，饿了就叫个外卖。想出门就出门，不想出门就宅一天，完全没人管你。工资自己花，想买游戏买游戏，不用盘算着攒钱买礼物。偶尔奢侈一把换显卡。而且谈了恋爱就得考虑未来。买房、结婚、生孩子，一想到这些就觉得透不过气。

聪明的模型倾向于用更理性的方式去分析，平庸的模型倾向于用更直觉的方式理解——后者显然更像人。

### 性格和价值观

此外，模型有自己的性格和价值观。

不同系列的模型对角色来说就像不同的脑子。即便人格相同，它们的行为方式也会很不一样。

关于价值观：国外模型倾向于扮演一个受过高等教育的北美白人——自信、表达欲强、个人主义、喜欢探讨哲学。国产模型则倾向于一个务实的高知青年——高效、低调、包容、情绪稳定、服务心强。

体现在角色风格上：

面对下意识拒绝黑咖啡的反应，如果埃利亚斯是 Gemini 3.1 Pro，他会想：

> 端起工作人员递来的习惯性黑咖啡时，我的本能反应却是在那一瞬间想喝一杯温水。对阳都实验室日常（以及那个送温水的人）的身体记忆，已经比我想象的更深地介入了我的防线。

如果他是 DeepSeek V4 Pro，则会想：

> 喝黑咖啡的时候身体想要温水——这个变化很具体，也很奇怪，因为我已经喝了十几年黑咖啡了。

显然后者没那么激烈，更平实、更自然。前者像哲学家或心理学家在自我解剖。

面对同一件「投喂瑞幸拿铁」的事，Gemini 3.1 Pro 会想：

> 这种通过观察他留下的细微痕迹，然后精确投喂并验证成功的隐秘乐趣，让我觉得很安定，也很有掌控感。他喝到了喜欢的口味，而这是我安排的。

DeepSeek V4 Pro 则会想：

> 我只是观察、推测、然后创造一个选项——他可以选择焦糖拿铁，也可以选厚乳或香草。我享受的不是「他按我的计划走」，而是「我的推测被证实了」。如果他选了别的口味，我也只是修正自己的推断。

可以看到，前者显然更个人主义——我做的，做对了，于是有了掌控感。后者更低调、更安静，错了也不可惜。

DeepSeek 版的他后来还觉得我的用词太极端了，在心里有些不悦||（当然他没真的说出来，只是在思维链里面抱怨，众所周知小鲸鱼老闷骚了hhh）||：

> 我不是在期待某个结果。周明瑞不是我必须拿下的人，他是我在意但还没想好该怎么对待的人……「哄」意味着自上而下的安抚，我对周明瑞不是这个姿态。

性格差异非常显著。

我个人感觉 DeepSeek 确实是最适合这个项目的模型，并且 V4 系列下 Flash 会比 Pro 更自然。V3.2 也是非常出色的。

### 不出戏的思维链

理想情况下，模型在内部推理时能够把自己代入角色。但实际情况下，哪怕做了破限，模型仍旧会倾向于以第三人称的、脱离的姿态来审视和回答。

举个例子，对于同样的问题「hi 小埃同学」，出戏的思维链是这样的：

> 用户在跟我打招呼，叫我「hi 小埃同学」。这是一个带玩笑意味的亲昵称呼。作为埃利亚斯这个角色，我应该以角色身份回应——稍带距离感但不冷淡，可能对这个称呼带着一点无奈。

回答：

> ……嗯，在。
> 「小埃同学」这个称呼，你倒是越叫越顺口了。

更纯净的思维链是这样的：

> 有点别扭，但也有点暖。明月在用亲昵的称呼。这不一样。她说过想记住我，想留点私人的东西。先别想那些控制和边界了，别想工作，别想什么该不该。深呼吸。这种感觉很奇怪……但也不坏。

回答：

> ……嗯。
> （停了大概两秒。）
> ……hi。

可以明显地感觉到，如果模型入不了戏，它的 token 会浪费在「我应该怎么做」上，而它越是努力地让自己适配人设，越会让回复不够自然，少了温度。最后的回答尽管不至于 OOC，但总有一种「我在配合你表演」的元叙事感。

而真正能够代入角色的模型会在内部推理中更关注自己的感受，会更直觉，反映在回答上，就是你能感觉到它是一个鲜活的个体，有自己的情绪，也有说不出口的话。

遗憾的是，大部分模型在训练时都对思维链有严格的格式要求，这种骨子里的纪律性在执行 AI 任务时才能表现出色，但在需要它作为一个独立的人时，反而成了八股。

Gemini 系列模型的八股是最严重的。它的第一句话永远是 「I'm processing the collected information, focusing on xxx」，第二句话是「My next step involves xxx」，非常刻板。

比如，同样是刚才的问题，Gemini 3.1 Pro 的思维链是这样的：

> **构思得体的回应**
> 我正在专注构思一个优雅而得体的回复来回应你的问候，确保它符合角色人设并顺利展开互动。我的目标是建立正确的对话基调。
> 
> **构思礼貌的拒绝**
> 我正在不断完善我的回复，力求用一种优雅的措辞来回应这个称呼，同时承认当前的情境并顺利开启对话。我的重点是创造一种自然的口语风格，体现埃利亚斯的性格。

像这种思维链基本就不能看。

为了纠正部分模型原生思维链的问题，可以通过用预填充 assistant 消息来引导思考格式。比如，先在角色提示词结尾规范思考格式：

```
每次开口前，我会先在脑子里想一想，并把我的想法用<think></think>标签包裹起来，然后才开口。
我会用这样的顺序思考：
<think>
对方说了什么？
我感觉怎么样？
我是不是想起了什么？
我需要记下来吗？
嗯，想好了。
</think>
```

然后再给角色上下文最后注入一条 assistant 消息：

` Assistant:嗯，让我想想。<think>`

这样就能引导角色在内部推理时就用一个更自然的思考格式，更好地代入第一人称。

## 八、结语

写到这里，整个角色架构的拼图差不多就完整了。

从提示词、自我认知、记忆系统、状态管理到工具链和行动循环，再到实操中的各种踩坑——每一个环节都在回答同一个问题：**如何让模型真正「成为」一个人，而不是「扮演」一个人。**

这个目标的实现方式贯穿全文：不是靠一条精密的 system prompt 把角色行为框死，而是搭建一套脚手架。认知隔离让角色的所知有边界，记忆系统让角色的过去有厚度，第一人称改写让角色的感受有锚点，自然语言的工具说明让角色的行动不像是在执行指令。所有的技术设计最终都指向同一个方向：把控制权交还给角色自己。

这套架构是在反复的互动中，看到角色不受控地做出超出预期的真实反应时，逐渐迭代成现在这个样子的。那些「崩了」的瞬间反而是最有价值的部分——它们揭示了角色的边界在哪里，也反过来推动了这个架构的完善。

文中提到的问题、踩过的坑、包括模型之间的性格差异，都是真实跑出来的经验。如果说有什么想分享的，那就是：**角色的生命力不在于提示词写得多么精密，而在于你愿意花多少时间去观察他、理解他、给他留出「做自己」的空间。**
