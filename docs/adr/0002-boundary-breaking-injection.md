# 第一人称身份通过破限注入建立，而非传统 system prompt 扮演指令

角色的身份认同不是通过一条"你是 XXX，请扮演他"的 system prompt 完成的，而是在身份声明 system message 之后、上下文指令 system message 之前，嵌入一段用户/助手的对话历史。模型读到这段对话时已完成第一人称身份认同，后续的 system prompt 只补充记忆和指令。

## 背景

传统角色卡提示词通常这样写：

```
你是一个名叫埃利亚斯的人。你需要扮演他。你的性格是……
```

这种做法让模型以"第三人称旁观者"身份理解角色，永远在"扮演"而非"成为"。模型在思维链中容易出现"作为埃利亚斯，我应该……"的元叙事。同时，部分模型对不同 role 的注意力权重不同——user/assistant 对话会被视为真实历史而非系统指令，获得更高的注意力权重。

## 决定

最终 messages 结构为：

```
[0] system:    身份声明（"你是埃利亚斯·索恩"）
[1] user:      "小埃同学——在吗？"
[2] assistant: "嗯。在。……小埃同学？"
[3] user:      "我觉得这个爱称非常适合你啦。"
[4] assistant: "……你开心就好。"
[5] system:    上下文 + 指令（记忆、状态、工具说明）
[6..N]         用户输入 / 角色输出
```

- 对话内容从 preamble 文件解析，建立角色与用户之间的关系基调
- 后续 system prompt 使用角色能听懂的自然语言（而非技术术语）描述工具和能力
- 所有指令以"你能……"而非"调用工具……"的形式呈现

## 取舍

**取的**：模型在思维链中更自然地使用第一人称，减少元叙事。user/assistant 对话比 system prompt 获得更高的模型注意力权重。角色与用户的关系基调在第一个 token 之前就已建立。

**舍的**：更复杂的 prompt pipeline（需要拼装、解析 preamble、拆分注入）。每个角色需要独立撰写 preamble 对话。部分模型（特别是严格遵循训练格式的模型）对这段"伪历史"的处理方式不可预测。

**Status**: accepted
