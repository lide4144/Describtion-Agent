# 故事包作为产出单元，用户仅通过 GM 交互

系统产出从「离散的角色蓝图」改为「完整的故事包」。用户不直接与角色交互，而是通过 GM 间接影响故事。用户在叙事中是不可见的——角色不知道用户的存在。

## 背景

最初的架构以「角色」为产出单元——铸造师每次生成一个角色蓝图，运行时引擎加载后让角色直接与用户对话（Chat 模式）。但随着设计深化，几个核心问题浮现：

1. **多角色故事的共享世界设定**需要一个统一载体，离散的角色蓝图无法承载「所有角色都知道的公共世界观」
2. **GM 需要知情**——它需要知道角色名单、世界设定、叙事风格才能安排场景
3. **用户在叙事中的身份不清晰**——直接跟角色说话时，角色是知道用户在"跟他聊天"还是不知道？
4. **用户的自角色需求**——用户想要一个故事内的化身，但这个化身不应该是用户直接控制的傀儡

这些问题的共同答案指向同一个方向：**铸造师的产出应该是一个完整的故事包，而不是零散的角色文件。**

## 决定

### 故事包结构

铸造师产出 `pi-characters/{story-name}/` 目录，包含：
- `story.yaml` — 共享世界观 + 角色名单 + 认知边界
- `gm.yaml` — GM 叙事风格 + NPC 配置
- `chars/*.yaml` — 各角色蓝图（含可选的 `user.yaml` 自角色）
- `saves/` — 运行时存档（写入时创建，非铸造产出）

系统代码与故事数据完全解耦。代码不包含任何故事特定的内容。

### 用户-GM-角色的交互模型

```
用户（隐形上帝）
  │  只跟 GM 对话
  ▼
GM
  │  理解意图 → 布置场景 → 驱动角色
  ▼
角色 agent(s) —— 不知道用户存在
```

- 用户不直接与角色对话
- 用户不知道角色的 private 状态（满足认知隔离）
- 角色不知道用户在「观看」
- GM 是用户和故事之间的屏障

### 用户自角色（可选）

当用户在铸造阶段选择创建自角色时：
- 铸造师将自角色作为一个普通角色蓝图加入故事包（`chars/user.yaml`）
- 自角色与故事角色共享同一套运行时机制
- 用户通过潜意识式指令影响自角色——通过 GM 传达意图，自角色自主决定是否采纳
- 自角色可以不同意用户的想法，就像其他角色可以不同意一样

### 存读档依托 pi session 系统

- 每个故事使用独立的 pi session 文件（`.jsonl`）
- 命名存档 = session 树的标签 + 记忆快照（custom entry）
- 故事切换 = 关闭当前 session，打开另一个
- 隔离通过 session 文件 + Nocturne Memory namespace 双重保障

## 取舍

**取的**：
- 用户与角色之间的认知隔离更纯粹（用户不在叙事内）
- 故事包自包含，便于分享和版本管理
- GM 可以统筹场景安排，故事体验更连贯
- 自角色的自主性让「扮演」更有沉浸感

**舍的**：
- 用户不能直接跟角色"聊天"（需要 GM 作为中介）
- 自角色的潜意识式控制需要用户适应（不能直接下命令）
- 实现复杂度更高（需要 GM agent + 多角色 agent 协调）
- 存读档需要同步所有 agent 的状态

**Status**: accepted
